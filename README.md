# 🐟 水産加工経営ゲーム（Webアプリ版）

ソニーの「戦略MG（マネジメントゲーム）」をベースにした、水産加工会社の経営ボードゲーム。
魚を**仕入れ→加工→販売**し、20ターン（4季節×5期）後の純資産で勝敗を競う。

**ローカル対戦（pass-and-play）** と **オンライン対戦（P2P）** の両方に対応。
ゲームロジックは純粋TypeScriptの `engine`、通信は差し替え可能な `net`（Transport抽象）に分離してある。

## 動かし方

```bash
npm install
npm run dev      # 開発サーバー（http://localhost:5173）
npm test         # エンジン＋同期ロジックのユニットテスト
npm run build    # 本番ビルド
```

## オンライン対戦（P2P / PeerJS）

メインメニューから:
- **オンライン部屋を作る** → 発行された**部屋コード**を相手に共有 → 全員そろったら「ゲーム開始」（空席はCPU）
- **部屋に参加** → 会社名と部屋コードを入力して参加

仕組み：**ホスト権威モデル**。ホストが`GameState`を保持してengineを実行（CPUもホスト側）。各プレイヤーは自席の`Command`をホストへ送り、ホストが適用して全員に新stateを配信するので同期ズレが起きない。署名はPeerJSの公開ブローカー（アカウント不要）。通信は`src/net/`のTransport抽象なので、将来Supabase/PartyKit等へ差し替え可能。

> 実際に別端末と遊ぶには下記の**公開（GitHub Pages）**が必要です（ローカルdevでも同一PCの2タブで動作確認は可能）。

## 公開（GitHub Pages）

`.github/workflows/deploy.yml` を同梱。手順：
1. このフォルダ（`suisan-kako-game`）を**単独のGitHubリポジトリ**として push（main/master）。
2. リポジトリの **Settings > Pages > Source** を「**GitHub Actions**」に設定。
3. push するたびに自動ビルド→Pages公開。発行URLを相手に共有すればオンライン対戦できる。

（`vite.config.ts` は本番 `base: './'` 済みなのでサブパス配信でも動作します。）

## ゲームの流れ（1ターン）

1. **シケ・市場** — 季節別確率でシケ具合（無風/風強め/台風）を抽選。漁獲量＝グレード基準量 × シケ倍率（◎=1/0.5/0.25、◯=0.5/0.25/0、△=0.25/0/0）。
2. **朝の準備（セリ前）** — 仕入れ前に**配置転換**（営業↔製造を移動）と**解凍**（冷凍庫→原魚）。※解凍した原魚はその日に加工しないと翌朝**腐って消える**。
3. **朝のセリ** — 全社が封印入札（**単価/kg と 欲しい数量kg** を指定）。**高単価から順に数量を割り当てる**ので、1つの魚を複数社で分け合えます。締切後に**「🔨 セリ結果発表」画面**で割当内訳を発表（一呼吸）。販売成立時は**「💰 販売成立」ポップアップ**。
4. **操業** — 手番制。各社は **その日「製造」か「販売」のどちらか一方だけ**（戦略MGと同じく1日1アクション）。選んだら本日の操業は終了：
   - **製造**（自分だけ・他社は割り込めない）原魚→製品に加工（歩留まり pc/kg・その日の製造枠ぶん）。
   - **販売**（他社も相乗り参加できる。相乗りは自分のアクションを消費しないので、製造した日でも相乗りで売れる。販売量は営業キャパで制限）以下の3販路へ：
     - **中央市場**＝原魚専用・無制限・固定価格（魚価イベントで変動）
     - **スーパー**＝加工品・販路キャパpc（全社共有・出荷で価格逓減）
     - **EC・輸出**＝加工品・HACCP認証で解禁（EU HACCPで+20%）
   - **パス**。全員が連続パスでこの日のアクション終了。任意で**冷凍**（原魚を保存。解凍は翌朝の「朝の準備」で）。
5. **決算（5期ごと）** — 人件費・固定費・借入利息を精算。各社が**内部環境カード**を1枚、全社共通で**外部環境カード**を1枚引く。
6. **投資（5期ごと）** — 在庫拡張・製造ライン増設・研修・採用・認証取得・借入。

## 環境カード

- **外部（30枚）**：魚価・漁獲量・スーパー販路キャパを ×2 / ×1/2（魚種別 or 全魚種）。次の四半期に作用。
- **内部（8枚）**：採用/退職/無料教育/教育無効/無料設備拡張/製造設備故障/在庫故障/ストライキ。各社が決算時に1枚。

## 戦略の軸

- **高付加価値型**：加工品強（茶漬け・たたき・味噌煮・白子）＋認証でEC輸出、高単価で売る。
- **大量回転型**：安く大量に仕入れ、原魚を中央市場で回転させてキャッシュを回す。
- スーパーは**全社共有キャパで出荷するほど値崩れ**するので、相乗り販売の駆け引きが効く。

## アーキテクチャ

```
src/
  engine/            純粋TSのゲームロジック（UI非依存・テスト対象）
    types.ts         ドメイン型
    data.ts          マスタ＆バランス定数（★ここを調整）
    random.ts        シード付き乱数（再現性のため）
    engine.ts        createGame() と applyCommand() ＝公開API
    engine.test.ts   ユニットテスト
  ui/                Reactコンポーネント（engineを表示・操作するだけ）
  App.tsx            画面オーケストレーション
```

**設計方針**：UIは `engine` の `GameState` を表示し、`Command` を `applyCommand(state, cmd)` に渡して
新しい状態を受け取るだけ。これによりオンライン化（サーバーで `applyCommand` を回す）への移行が容易。

## データの出どころと未確定点

魚種・季節グレード・仕入/製品価格・歩留まり・販路キャパ・環境カードは**スプレッドシート本データ準拠**。
人件費・固定費・投資額・借入額・初期資本に対する価格スケールは**調整値**（`data.ts` 冒頭にまとめてあり要バランス調整）。

未確定（スプシから解釈しきれず、確認したい点）：
- **原価率**（加工品弱 0.3〜0.6 / 強 0.2〜0.4）：仕入れをセリで別途支払うため二重計上回避で**現状未適用**。加工コストとして使うか要相談。
- 製品表の **スーパー（kg）列**（10/7/8/5…）の意味が不明。現状はスーパー(pc)=30をキャパに採用。
- 内部カードの「採用/教育」の**対象（営業か製造か）**は暫定ルールで自動選択。

## 画面の主な機能

- **CPU（AI）対戦**：セットアップで各社を人間/CPUに切替。1人でCPUと対戦・全員CPUの自動対戦も可（`src/engine/ai.ts` のヒューリスティック）
- **会社盤（プレイヤーマット）**：工場・冷凍庫・営業所ゾーン、人の駒、販路ピル、在庫ゲージ
- **フェーズ進行ステッパー**＋**季節テーマ**（季節ごとにアクセント色が変化）
- **スーパー値崩れメーター**：販路の充填度＝価格下落を可視化
- **環境カードのバナー演出**（外部）＋**各社の内部カード一覧**（決算時）
- **純資産の推移チャート**（終了画面）
- **オートセーブ＆再開**（localStorage・リロードしても続行）
- **遊び方ヘルプ**（魚種・季節・販路・製品の早見表）
- **発表演出**：セリ結果リビール／販売成立ポップアップ／決算の損益めくり／優勝スポットライト（紙吹雪）
- **効果音**（Web Audio合成・木槌/コイン/決算/ファンファーレ・🔊ミュート切替）
